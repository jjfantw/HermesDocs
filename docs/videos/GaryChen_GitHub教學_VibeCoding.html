01

GitHub 與 Git 的差別

1:02

GitHub 是程式碼專用的雲端儲存平台，類似 Google Drive 但專為程式碼設計，讓有權限的人可以隨時查看甚至一起編輯。

Git 則是在本機端幫資料夾建立版本紀錄的工具。GitHub 不收單純的檔案，只接收已經做好版本紀錄的專案。

用 `git init` 指令告訴 Git 開始追蹤這個資料夾的所有變動。

基礎概念

📋 本節指令一覽

  1. git init
02

Commit：建立安全檢查點

3:21

Commit 就像玩遊戲打大魔王之前的手動存檔。它幫程式碼建立一個絕對安全的檢查點，只要做了 Commit，這份正常狀態就被永久封存。

Commit 不是上傳，它只是在自己電腦裡幫目前的狀態留下一份紀錄。之後就算 AI 把程式碼全刪了或改亂，都可以瞬間還原回 Commit 時的完美狀態。

當 AI 問「要不要 Commit」，它其實是在問：現在這個好好的狀態，你要不要先存個檔、買個保險？

版本管理

📋 本節指令一覽

  1. git commit -m "訊息"
03

Push：把存檔送上雲端

3:21

Push 才是真的把電腦裡存好的檔案推送到雲端的 GitHub 上。Commit 跟 Push 完全是兩回事。

第一次 Push 前必須先設定 Remote（遠端地址，即 GitHub 上的專案資料夾），通常預設名稱為 Origin。Main 代表專案目前的官方正式版本。

Push 到 Origin Main 的白話文：請幫我把現在寫好的正式版本，傳送到雲端的那個專案資料夾裡面。

⚠️ Push 前一定要確認機密檔案（API 金鑰、密碼等）已透過 `.gitignore` 排除，否則金鑰外洩就像把大門鑰匙公開貼在網路上。

雲端備份

📋 本節指令一覽

  1. git push origin main
  2. git remote add origin <URL>
04

Clone 與 Pull：協作同步

6:11

Clone 是把 GitHub 上的專案完整複製到自己的電腦，包含完整的 Git 追蹤紀錄。這跟網頁上的「Download ZIP」完全不同——ZIP 下載的只是純粹檔案，沒有版本紀錄，之後無法把改動同步回 GitHub。

Pull 是把 GitHub 上面最新版本的程式碼抓下來，同步到自己的電腦。

協作的基本循環：第一次加入專案用 Clone → 平常同步別人進度用 Pull → 自己改好東西先用 Commit 存檔 → 最後用 Push 送上雲端。

協作流程

📋 本節指令一覽

  1. git clone <URL>
  2. git pull origin main
05

Branch：隔離開發風險

6:11

Branch（分支） 是 Git 裡非常關鍵的概念。所有專案一開始都有一條主線叫 Main，代表穩定運行的程式碼。

開 Branch 的目的是讓正在開發中、還不穩定的功能，跟已經寫好很穩定的功能徹底分開。

可以把專案資料夾想像成一張神奇辦公桌，Branch 就像桌子旁邊的平行時空切換按鈕。按 Main 看到穩定版本，按新功能 Branch 看到開發中的草稿。

如果新功能被 AI 改爛了，最壞的狀況就是把這個 Branch 直接刪掉，切回 Main，Main 依然毫髮無傷。

開發新功能的第一步：請 AI 開一個 Branch，接下來所有的改動和 Commit 都在這個 Branch 上做，絕對不要動到 Main。

風險隔離

📋 本節指令一覽

  1. git checkout -b <branch-name>
  2. git branch
  3. git checkout main
06

Worktree：多 AI 平行開發

10:42

Branch 的致命傷：雖然可以切換不同狀態，但同一時間只有一張辦公桌（一個資料夾）。如果兩個 AI 同時做事，一個要看資料庫藍圖、一個要看 UI 設計稿，他們會為了搶桌子而大打出手。

Worktree 就是直接去 IKEA 再買第二張實體辦公桌。Branch 是在同一張桌子上切換時空，Worktree 則是直接給你第二張實體的桌子，讓兩個時空同時並存在電腦裡。

AI 會生出兩個獨立的資料夾，但這兩張桌子連通同一個 Git 記憶庫。讓 Agent A 去第一張桌子負責資料庫 Branch，Agent B 去第二張桌子負責 UI Branch，平行開發效率直接拉滿。

以前的工程師不太用 Worktree（一次專心做一件事），但 Vibe Coder 會同時開兩三個 AI Agent，就一定會需要 Worktree。

多工開發

📋 本節指令一覽

  1. git worktree add <path> <branch>
  2. git worktree list
07

PR 與 Merge：合併審核流程

10:42

PR（Pull Request） 是一份改動提案。功能寫完後，開 PR 把 Branch 上的改動送到 GitHub 上讓大家一起審核。

可以直接請 AI 代勞開 PR，內容說明做了什麼功能、改了哪些檔案、要怎麼測試。

審核通過後，按下 Merge 按鈕把程式碼正式合併回 Main 主線。

⚠️ 合併是在雲端網頁上發生的，自己電腦本機的 Main 不會自動同步。每次有新功能被合併進 Main 後，一定要切回本機的 Main 再執行一次 Pull，手動把雲端最新版本拉下來。

協作流程

📋 本節指令一覽

  1. git merge <branch-name>
08

Conflict：衝突處理

10:42

當兩個人改到同一個檔案的同一個地方，Git 會跳出 Conflict（衝突） 警告。Git 發現同一段程式碼出現兩個不同版本，不知道該聽誰的。

現代 AI 工具（如 Claude Code）遇到 Conflict 時會自己判斷處理，通常有三種方式：

① 二選一：AI 判斷哪一版才是正確的邏輯，直接保留正確的那一邊。

② 兩邊都留：兩人剛好都在最後一行新增不同項目，Git 死板判定為衝突，但 AI 看得懂其實只是新增不同項目，會聰明地把兩個都保留下來。

③ 打掉重寫：兩邊邏輯硬拼在一起會出錯，AI 看懂雙方目的後，乾脆把那段程式碼重寫一遍，把兩邊功能順順接在一起。

如果 AI 卡住，Vibe Coder 要做的不是自己下去改程式，而是給出明確的產品決策方向，AI 就會自己挑選最適合的合併方式。

09

改壞了怎麼救：Restore 與 Revert

15:53

情境一：還沒 Commit — AI 剛改了一大堆檔案，跑了一下發現完全不對。對應的動作是 Restore，一鍵讀檔回到上一次的遊戲存檔點，當作剛剛 AI 亂寫的事情完全沒發生過。

情境二：已經 Commit 了 — 順手存檔後才驚覺那次 Commit 是錯的。安全的補救作法是 Revert。Revert 不是偷偷刪掉歷史紀錄，而是新增一個反向的 Commit，去把前面那次錯誤的改動抵消掉。

Revert 在跟朋友合作專案時特別安全，因為所有人都能在歷史紀錄裡看到「有加錯的東西 → 又被抵消掉」的完整過程。

這些指令都不用自己硬背。發現 AI 改壞時，直接跟 AI 說「剛剛這次的改動我不要了，請幫我回到上一個正常的版本」，AI 就會自己判斷該用 Restore 還是 Revert。

救援方法

📋 本節指令一覽

  1. git restore <file>
  2. git revert <commit-hash>

你不需要在今天逼自己變成 Git 專家，但你要知道，當 Claude 或 Cursor 問你要不要 commit、要不要開 branch、要不要發 PR 的時候，他其實是在問你：這次的開發你要不要先存檔？你要不要開個空間來隔離風險？你要不要把寫好的東西交出去給別人檢查？只要這些判斷你聽得懂，你用 AI 開發就會踏實很多。
